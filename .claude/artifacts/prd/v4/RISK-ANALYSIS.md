## 1. 리스크 요약 (Executive Summary)

### 1.1 핵심 메시지

Maththera PRD v4에서 도출된 리스크는 총 **12건의 명시적 리스크**, **6건의 에이전트 간 충돌**, **21건의 미해결 질문**으로 구성된다. 이 중 **높음(High) 등급 리스크가 8건**(67%)으로 전체 리스크 프로파일의 과반을 차지하며, 특히 RT 기준값 미검증(RSK-001), Probe 분기 로직 불완전(RSK-002), DTx 인허가 불확실성(RSK-008), 아동 데이터 프라이버시(RSK-011) 등이 MVP 출시 전 반드시 해소해야 할 블로킹 리스크이다.

추가로, 에이전트 아키텍처 선택(단일 vs 다중)은 PRD v4에서 명시적으로 결정되지 않은 상태이며, 이 결정이 제품/기술 리스크 7건 중 5건의 완화 전략 실행에 직접 영향을 미친다.

## 2. 제품/기술 리스크 상세

### RSK-001: RT 기준값 미검증

| 항목 | 내용 |
|------|------|
| **설명** | 12초(느림), 0.8초(극빠름) 등 핵심 임계값이 실측 데이터 없이 경험적으로 설정됨. 유스케이스 1(전략 불일치)의 전체 진단 로직이 이 값에 의존 |
| **영향** | 잘못된 기준으로 정상 아동을 난산증으로 오진단하거나, 난산증 아동의 전략 미숙을 놓칠 수 있음 |
| **영향도** | 높음 |
| **발생확률** | 높음 -- 연령(만 5~10세), L-level, Cur-C level별 RT 분포가 상이할 것이 확실하며 단일 기준은 반드시 오차를 발생시킴 |
| **완화 전략** | 1) MVP 전 50명+ 파일럿에서 L-level x Cur-C level별 RT 기준 실측 <br>2) 기본 Fluency 임계값 5초 채택 (tech/CLM-010) <br>3) 문제 카테고리별 보정 계수 적용 <br>4) 파일럿 후 기준값 업데이트 파이프라인 구축 |
| **잔여 리스크** | 50명 파일럿 표본이 충분하지 않을 수 있음. Phase 2에서 대규모 데이터 기반 재보정 필요 |
| **담당** | PM + Research (파일럿 설계), Tech (측정 인프라) |
| **출처** | biz/RSK-001, pm/RSK-002 |

### RSK-002: Probe 분기 로직 불완전

| 항목 | 내용 |
|------|------|
| **설명** | 유스케이스 4(변인 통제 Probe)에서 두 가지 이상의 오답 원인이 동시에 존재할 때 판정 기준이 부재. Case A/B 이진 분기만 정의되어 있고, Case A+B 동시 해당이나 '판정 불가' 상태에 대한 처리가 없음 |
| **영향** | 복합 원인 아동에 대해 한쪽 원인만 진단하여 불완전한 훈련 경로를 처방. 진단 정확도 KPI(85%+) 미달 가능 |
| **영향도** | 높음 |
| **발생확률** | 높음 -- 난산증 아동의 약 30~40%가 복합 원인을 보이는 것으로 추정 |
| **완화 전략** | 1) 상태 머신 + 룰 엔진 정형화 <br>2) Probe Case A/B 단위 테스트 100% 커버리지 <br>3) '미정(UNDETERMINED)' 상태용 추가 Probe 전략 구현 -- 2차 Probe 문제로 남은 변인 재통제 <br>4) 3회 Probe 후에도 미정이면 전문가 수동 리뷰 큐로 이관 |
| **잔여 리스크** | 2차/3차 Probe 문제가 아동의 피로도를 증가시킬 수 있음. 세션 길이 제한(OQ-003)과 상충 가능 |
| **담당** | Tech (FSM 구현), PM (Probe 시나리오 정의), Research (감별 정확도 검증) |
| **출처** | biz/RSK-002, pm/RSK-001 |

### RSK-003: AI 피드백 톤 가이드 위반

| 항목 | 내용 |
|------|------|
| **설명** | 15자 초과, 힌트 2개 이상, 금지 표현 사용 등 톤 가이드를 위반하는 피드백이 생성될 수 있음 |
| **영향** | 수학 불안이 높은 아동에게 부정적 경험 제공. 세션 완료율 저하, 재방문율 하락. DTx 전환 시 치료적 재현성 훼손 |
| **영향도** | 중간 |
| **발생확률** | 높음 -- 동적 변수(숫자/연산자) 삽입 시 글자 수 초과 빈발 가능. 진단 질문과 피드백의 톤 가이드 충돌(CONF-004) 미해결 |
| **완화 전략** | 1) 사전 검수 템플릿 풀(상황별 20개+) 구축 <br>2) 글자수/문장수/금지어 후처리 검증 레이어(정규식 + 블랙리스트) <br>3) LLM 자유 생성 완전 차단 <br>4) QA 자동화: 모든 템플릿 조합을 CI/CD에서 검증 |
| **잔여 리스크** | 템플릿 풀 규모가 부족하면 응답이 반복적이 되어 아동 경험 저하. 피드백 다양성과 안전성 간 트레이드오프 |
| **담당** | PM (템플릿 설계), Tech (필터 구현), Research (톤 검증) |
| **출처** | pm/RSK-003, tech/RSK-003 |

### RSK-004: 4개 유스케이스로 실제 오답 패턴 미커버

|------|------|

### RSK-005: 진단 질문 응답 분류 오류

| 항목 | 내용 |
|------|------|
| **설명** | Adaptive Logic의 진단 질문(예: "어디부터 더했어?")에 대한 아동 응답을 분류할 때 오류 발생 가능. 특히 초등 저학년의 비정형적 표현, 선택지 오해, 무작위 선택 등 |
| **영향** | 잘못된 경로 분기(Route A를 타야 할 아동이 Route B로 분기). 진단 정확도 전체에 영향 |
| **영향도** | 높음 |
| **발생확률** | 높음 -- 초등 1~2학년 아동의 메타인지 능력 한계로 자신의 풀이 전략을 정확히 보고하기 어려움 |
| **완화 전략** | 1) MVP에서 객관식(2~3개 선택지) 채택으로 입력 오류 최소화 <br>2) 선택지 디자인: 오답 선택지를 진단적으로 유의미하게 구성(각 선택지가 특정 오답 원인을 시사) <br>3) Phase 2에서 STT+NLU 확장 <br>4) 응답 일관성 검증: 동일 유형 질문 2회 제시로 일관성 확인 |
| **잔여 리스크** | 객관식 자체의 한계 -- 찍기(무작위 선택)로 인한 오분류. 2~3지선다에서 찍기 정답률 33~50% |
| **담당** | Tech (입력 UI), PM (선택지 설계), Research (분류 정확도 검증) |
| **출처** | tech/RSK-001 |

### RSK-006: Cog-C / Cur-C 명명 충돌

| **완화 전략** | 1) Cog-C(인지진단) / Cur-C(교육과정) 접두사 분리 즉시 적용 <br>2) 데이터 스키마에 enum 타입으로 강제 <br>3) 코드 리뷰 체크리스트에 명명 규칙 검증 항목 추가 <br>4) 기존 소스 문서 일괄 업데이트 |
| **잔여 리스크** | 팀원 온보딩 시 기존 문서와 새 명명 규칙의 혼동. 일정 기간 과도기 비용 |
| **담당** | Tech (스키마 구현), Pedagogy (문서 업데이트), PM (체크리스트 관리) |
| **출처** | pedagogy 리스크, CONF-002 |

## 3. 비즈니스/GTM 리스크 상세

### RSK-008: DTx 인허가 12~24개월 불확실

| 항목 | 내용 |
|------|------|
| **설명** | 디지털 치료제(DTx) 인허가는 12~24개월 소요되며, 한국 규제 환경의 불확실성이 높음. 규제 기준 변경, 임상 데이터 요구사항 변동 가능 |
| **영향** | Phase 3 수익 모델(세션당 50,000원+, 건강보험 적용)의 실현 시점 지연. 투자 유치 스토리에 불확실성 증가 |
| **영향도** | 높음 |
| **발생확률** | 중간 -- 한국 식약처의 DTx 가이드라인이 아직 성숙 단계에 있음 |
| **완화 전략** | 1) 에듀테크 우선 출시 후 DTx 병행 추진(투트랙 전략) <br>2) Phase 2부터 임상 데이터 수집 인프라(세션 로그, 사전/사후 평가, 동의 관리) 구축 <br>3) 규제 전문가 자문 조기 확보 <br>4) DTx 선례 기업(Akili Interactive 등) 벤치마킹 |
| **잔여 리스크** | 에듀테크 모델만으로는 세션당 50,000원+ 가격 정당화 불가. DTx 무산 시 수익 모델 재설계 필요 |
| **담당** | Biz (전략), Legal (규제 자문), Tech (데이터 인프라) |
| **출처** | biz/RSK-003 |

### RSK-009: 난산증 인지도 낮음

| 항목 | 내용 |
|------|------|
| **설명** | 난산증(Dyscalculia)의 사회적 인지도가 난독증(Dyslexia) 대비 현저히 낮음. 학부모 대부분이 자녀의 수학 어려움을 '노력 부족'이나 '지능 문제'로 오인 |
| **영향** | B2C 고객 획득 비용(CAC) 증가. '난산증'이라는 단어 자체에 대한 저항감으로 전환율 저하 가능 |

## 4. 에이전트 아키텍처 리스크

Maththera의 핵심 AI 시스템인 Cogassist Agent의 아키텍처 선택은 제품/기술 리스크의 완화 실행에 직접 영향을 미치는 구조적 의사결정이다. 두 가지 후보 아키텍처를 기술적 관점에서 비교 분석한다.

### 4.1 Option A: 단일 에이전트 (Monolithic) 리스크

**구조**: 하나의 AI 에이전트가 진단(문제 출제, RT 수집), 분석(오답 원인 분류, Route A/B 결정), 튜터링(피드백 생성, 시각 보조)을 모두 수행.

| ID | 리스크 | 영향도 | 발생확률 | 설명 |
|----|--------|--------|---------|------|
| ARCH-A01 | **프롬프트 폭발 (Prompt Bloat)** | 높음 | 높음 | W-Score 6차원 규칙, C-Tag 15단원 + Cog-C 6태그 매핑, Adaptive Logic 4패턴 분기, 톤 가이드(금지어 목록 포함), Probe 분기 로직, 3단계 복귀 루프, Bridge Loop, White-out 처리 등을 단일 프롬프트에 포함해야 함. 예상 프롬프트 길이: 8,000~15,000 토큰 |
| ARCH-A02 | **컨텍스트 윈도우 고갈** | 높음 | 중간 | 세션 진행에 따라 누적되는 컨텍스트: 원래 문제, 단순화 문제, 각 단계 결과, W/C 위치, Probe 이력, 피드백 이력 등. 10문제 이상 진행 시 시스템 프롬프트(~15K) + 세션 컨텍스트(~5K/문제)로 컨텍스트 윈도우 한계 도달 가능 |
| ARCH-A03 | **단일 장애점 (Single Point of Failure)** | 높음 | 중간 | 에이전트가 하나의 추론 체인에서 진단/분석/피드백을 모두 처리하므로, 한 단계의 오류(예: RT 분류 실수)가 전체 파이프라인을 오염시킴. 오류 격리 불가 |
| ARCH-A04 | **규칙 간 간섭** | 중간 | 높음 | 톤 가이드의 "3문장/10자" 제약이 Adaptive Logic의 진단 질문 생성과 충돌(CONF-004). 단일 프롬프트에서 상충하는 규칙이 공존하면 비결정적 동작 발생 |
| ARCH-A05 | **테스트 불투명성** | 중간 | 높음 | 전체 시스템이 하나의 블랙박스로 동작하여, 진단 오류의 원인이 RT 분류인지, 오답 패턴 매칭인지, 경로 분기인지 특정하기 어려움. 디버깅 비용 증가 |
| ARCH-A06 | **업데이트 결합도** | 중간 | 중간 | 톤 가이드만 변경하려 해도 전체 프롬프트를 재검증해야 함. W-Score 규칙 추가/수정 시 피드백 품질에 예상치 못한 영향 가능 |

### 4.2 Option B: 다중 에이전트 오케스트레이션 리스크

**구조**: 3개 전문 에이전트(진단 에이전트, 분석 에이전트, 튜터 에이전트)가 오케스트레이터를 통해 협업.

- **진단 에이전트**: 문제 출제, RT 수집, 행동 태그(OBS_*) 관찰
- **분석 에이전트**: 오답 원인 분류, Route A/B 결정, Probe 분기
- **튜터 에이전트**: 피드백 생성, 톤 가이드 준수, 시각 보조 결정

| ID | 리스크 | 영향도 | 발생확률 | 설명 |
|----|--------|--------|---------|------|
| ARCH-B01 | **에이전트 간 통신 지연** | 높음 | 중간 | 진단->분석->튜터 순차 호출 시 각 에이전트 추론 시간 누적. NFR 요구사항 "파이프라인 2초 이내 E2E"(AC-001-3) 충족이 어려울 수 있음. 3번의 LLM 호출 vs 1번의 LLM 호출 |
| ARCH-B02 | **상태 동기화 복잡성** | 높음 | 높음 | 3단계 복귀 루프, Bridge Loop, Probe 이력 등의 세션 상태를 에이전트 간에 일관되게 공유해야 함. 상태 불일치 시 진단과 피드백이 서로 다른 문제를 참조하는 치명적 버그 발생 가능 |
| ARCH-B03 | **오케스트레이션 복잡도** | 높음 | 중간 | 진단-분석-튜터 간 호출 순서, 에러 핸들링, 폴백 전략, 타임아웃 처리 등 오케스트레이터의 로직이 복잡해짐. 오케스트레이터 자체가 새로운 단일 장애점이 될 수 있음 |
| ARCH-B04 | **비용 증가** | 중간 | 높음 | 문제 1개당 최소 3번의 LLM API 호출(진단/분석/튜터). Probe 발생 시 추가 호출. 단일 에이전트 대비 API 비용 2~4배 증가 예상 |
| ARCH-B05 | **책임 경계 모호성** | 중간 | 중간 | 분석 에이전트가 Route A/B를 결정했는데 튜터 에이전트가 부적절한 피드백을 생성한 경우, 원인이 분석 에이전트의 오분류인지 튜터 에이전트의 템플릿 선택 오류인지 경계가 모호해질 수 있음 |
| ARCH-B06 | **개발/운영 오버헤드** | 중간 | 높음 | 3개 에이전트 각각의 프롬프트 관리, 버전 관리, 테스트, 모니터링이 필요. 14주 MVP 일정(Phase 1) 내에 다중 에이전트 시스템을 안정화하기 어려울 수 있음 |

### 4.4 아키텍처 선택 권고

**권고: MVP는 Option A(단일 에이전트), Phase 2부터 Option B(다중 에이전트)로 점진 전환**

| Phase | 아키텍처 | 근거 |
|-------|---------|------|
