# Design Spec 작성 가이드 v3.0

AI 코딩 도구가 이모코그 표준 디자인 시스템을 준수하는 UI를 생성하기 위한 규칙 문서

- **핵심 질문:** "AI가 생성한 80%의 초안 위에, 디자이너가 20%의 인간적 판단을 더해 완성도를 높이고 있는가?"
- **프로세스 위치:** S4 Freeze 단계 핸드오프 문서
- **Gate 연결:** Gate 4 — PM·디자이너·엔지니어 3자 합의 후 S5 개발 진입
- **작성자:** AI 1차 생성 → 디자이너 주도 검토 및 확정

---

## Static vs Dynamic 계층 구조

Design Spec은 두 계층으로 구성된다.

| 계층 | 항목 | 성격 | 작성 주체 |
|------|------|------|----------|
| Static Layer | 1~8번 | 전사 공통 디자인 헌법. 모든 제품에 동일하게 적용 | 이모코그 표준값 (수정 불가) |
| Dynamic Layer | 9~11번 | Product Spec 기반 서비스별 커스터마이징 | AI 1차 생성 → 디자이너 확정 |

**Static Layer 원칙:**
- AI는 이 영역의 값을 임의로 수정할 수 없다.
- Dynamic Layer 규칙과 충돌 시 Dynamic 값을 기준으로 한다.
- 변경이 필요한 경우 Tech Spec 담당자와 협의 후 Freeze 체크리스트에 사유를 명시한다.

**Dynamic Layer 원칙:**
- AI가 Product Spec을 읽고 3단계 추론(컨텍스트 추출 → 파라미터 매핑 → 전략적 합성)을 통해 1차 산출물을 생성한다.
- 디자이너가 반드시 검수하여 최종 확정한다.

## Dynamic Layer 작성 상세

### 항목 9: CSS 변수

Static Layer 기준이 틀이 되고, 서비스 톤앤매너에 따라 아래 값만 수정한다.

- `--color-primary-500`: 브랜드 컬러 (OKLCH hue: 250=파랑, 145=초록, 25=빨강)
- `--foreground`: 기본 `0.208` → 고대비 타겟 시 `0.15`로 낮춤
- `--border`: 기본 `0.929` → 구분선 강화 필요 시 `0.85`로 낮춤
- `--shadow-color` 투명도: 기본 `0.1` → 입체감 강화 필요 시 `0.2`로 높임
- 시니어 타겟 추가: body에 `line-height: 1.6; letter-spacing: -0.01em;` 추가

예시 파일(`기억생생 Design Spec 예시.md`)의 CSS 변수에서 `[변경]` 주석이 달린 항목이 실제 수정값이다.

### 항목 10: UX 전략

Product Spec을 읽고 3단계 추론을 통해 작성한다.

```
1. 컨텍스트 추출: "이 서비스의 타겟은 누구이고, 핵심 가치는 무엇인가?"
   → Product Spec §7 (타겟 인사이트) 참조

2. 파라미터 매핑: "Static Layer의 어떤 가변 요소를 어떻게 조정해야 하는가?"
   → 접근성 기준(AA/AAA), 폰트 최소 크기, 인터랙션 속도, 터치 타겟 크기 등

3. 전략적 합성: "이 제품만의 UX 원칙은 무엇인가?"
   → 페인포인트 해소 방법, 정서 설계 전략, 신뢰 구축 방식
```

**출력 형식**: `# | 분류 | 항목 | 상세` 4열 테이블. 번호는 `분류번호-항목번호` (예: 1-1, 1-2, 2-1) 형식으로 작성한다.

| 분류 구성 | 예시 (기억생생) |
|----------|--------------|
| 분류 1: 인지·신체 | 기억 보조 UI, 고대비 시각 보정, Touch Guard, Step Navigation |
| 분류 2: 감각·피드백 | 다감각 실재감, 속도 제어, 맥락 유지 애니메이션 |
| 분류 3: 정서·신뢰 | 실패 피드백, AI 컴패니언, 성취 가시화 |

### 항목 11: Human Touch 체크리스트

**이 항목의 본질**: 코드 테스트나 자동화로 검증할 수 없는, 디자이너가 직접 서비스를 사용하며 '체감'해야 하는 감각·정서·존엄성 관련 항목이다.

**Product Spec에서 도출하는 방법:**

```
1. 타겟 사용자가 이 서비스를 사용할 때 가장 취약한 '정서적 순간'은 언제인가?
   → 처음 시작할 때 / 실패했을 때 / 보상을 받을 때 / 떠나려 할 때

2. 그 순간에 사용자가 느끼길 바라는 감정은 무엇인가?
   → 환영받음 / 존중받음 / 성취감 / 안도감

3. 디자이너가 직접 앱을 사용해보며 그 감정이 실제로 전달되는지 체크한다.
```

**출력 형식**: `# | 분류 | 항목 | 상세` 4열 테이블. 번호는 `분류번호-항목번호` 형식으로 작성한다.

| 분류 구성 | 예시 (기억생생) |
|----------|--------------|
| 분류 1: 언어·정서 | 동반자적 어법, 부정적 낙인 제거, 자존감 인터벤션 |
| 분류 2: 흐름·맥락 | 3초 컷 직관성, 힌트의 실질적 가치, 속도 리듬 |
| 분류 3: 감각·보상 | 보상의 쾌감, 시각적 품격, 햅틱 적절성 |

**작성 원칙:**
- "~인가?" 형식으로 디자이너의 주관적 체감을 묻는다.
- 기술 스펙(px, ms 등) 검증이 아닌 감각·감정 검증에 집중한다.
- 타겟 사용자를 구체적으로 떠올리며 작성한다 (예: "내 부모님이 이 앱을 쓴다면...").
