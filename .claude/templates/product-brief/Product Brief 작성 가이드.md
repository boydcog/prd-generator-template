# **Product Brief 작성 가이드 v1.0**

**Product Brief \= 프로젝트 나침반 \= AI의 시스템 프롬프트**   
이 문서 하나로 사람은 프로젝트를 이해하고, AI는 다음 산출물을 생성한다.

* **핵심 질문**: "이 아이디어가 풀 만한 문제인가, 누구를 위해 무엇을 만들 것인가?"  
* **프로세스 위치**: Stage 1 \- Brief  
* **Gate 연결**: Gate 1 (시장+사업성 Go/No-Go)  
* **작성**: PM (AI 80% 초안 \+ PM 20% 판단)  
* **데이터 수준**: 가설 기반 (PM 경험 \+ 2차 데이터)

## **0\. Context Dump (PM 입력용)**

이 섹션만 PM이 직접 채운다. 나머지는 AI가 초안을 생성한다. 키워드, 메모, 생각 조각이면 충분하다.

| 프로젝트명: 한 줄 설명: 타겟 고객 (누구?): 핵심 문제 (뭐가 아픈가?): 우리 솔루션 (어떻게 푸는가?): 왜 지금? (타이밍/트리거): 경쟁/대안 (지금 어떻게 해결하고 있나?): 성공 \= ? (6개월 후 어떤 상태?): 의사결정자 / PM: 제약 조건 (기술/비즈니스/규제): 기타 메모: |
| :---- |

**예시:**

| 프로젝트명: Maththera 한 줄 설명: 수학부진 아동용 AI 수감각 훈련 웹앱 타겟 고객: 수학 성취도 하위 20% 아동(만4세\~초2)의 학부모 핵심 문제: 연 600만원 사교육비에도 기초 연산 미해결. 매일 2시간 학습 전쟁 우리 솔루션: AI가 오답 원인(수감각 결핍) 분석 → 맞춤 훈련 경로 \+ 공감형 피드백 왜 지금: AI 초개인화 학습 가능 \+ 기초학력 지원 예산 50% 삭감으로 공교육 공백 경쟁/대안: 아이설렘수학(난산증 특화), 토도수학(유아), 수학학원(월 35만원, 단순 반복) 성공 \= 8주 후: 사용자 테스트 "도움됨" 60%+ / 주간 리텐션 50%+ 의사결정자 / PM: Benji (Go/Kill) / Lilly (실행) 제약 조건: 웹앱(태블릿/PC), 생성형 AI 필수, 8주 내 MVP, 만14세 미만 개인정보 보호 기타: 임상연구 2026.08 예정. 웹앱(태블릿/PC) 우선 |
| :---- |

## **1\. 프로젝트 개요**

| 항목 | 내용 |
| ----- | ----- |
| **프로젝트명** |  |
| **한 줄 정의** | (20자 이내. "누구를 위한 무엇") |
| **North Star Metric** | (단 하나의 핵심 지표) |
| **의사결정자** | (Go/Kill 판정권자) |
| **PM** | (실행 총괄) |
| **버전** | v0.1 (초안) |

**가이드:** "한 줄 정의"는 누구에게든 10초 안에 설명할 수 있는 수준이어야 한다. North Star Metric은 단일 지표다. 복수 지표는 나침반이 아니라 지도다.

**예시:**

| 항목 | 내용 |
| ----- | ----- |
| **프로젝트명** | Maththera |
| **한 줄 정의** | 수학부진 아동을 위한 AI 수감각 훈련 웹앱 |
| **North Star Metric** | 훈련 전후 연산 정확도·유창성 개선율 (%) |
| **의사결정자** | Benji |
| **PM** | Lilly |

## **2\. 문제 정의**

| 항목 | 내용 |
| ----- | ----- |
| **타겟 고객** | (구체적 페르소나. "\~한 사람") |
| **핵심 문제** | (고객이 겪는 구체적 고통. 1\~2문장) |
| **문제의 빈도/강도** | (얼마나 자주, 얼마나 아픈가) |
| **현재 대안** | (고객이 지금 어떻게 해결하는가) |
| **현재 대안의 한계** | (왜 기존 방법이 불충분한가) |

**가이드:**

* "문제가 실재하는가?"는 Gate 1의 핵심 판정 기준이다  
* 근거가 있으면 반드시 기재한다: \[추정\] vs \[데이터\]

**예시:**

| 항목 | 내용 |
| ----- | ----- |
| **타겟 고객** | 수학 성취도 하위 20% 아동(만4세\~초2)의 학부모. 연 600만원 사교육비 지출 중 |
| **핵심 문제** | 6개월 이상 교육해도 기초 연산 미해결. 어제 배운 내용을 리셋된 듯 잊어버림 |
| **문제의 빈도/강도** | 매일 2시간 학습 전쟁. 초3 수학 미달 25% → 9학년 포기율 49% \[데이터: 교육부 통계\] |
| **현재 대안** | 수학학원(월 35만원), 학습지, 교육앱(토도수학, 아이설렘) |
| **현재 대안의 한계** | 단순 반복(양치기) 방식. 수감각 결핍이라는 근본 원인에 개입하지 못함 \[데이터\] |

## **3\. 타이밍**

| 항목 | 내용 |
| ----- | ----- |
| **왜 지금?** | (시장 변화, 규제, 기술, 경쟁 등 트리거) |
| **기회의 창** | (이 기회가 언제까지 유효한가) |

**가이드:** "왜 6개월 전이 아니라 지금인가? 왜 1년 뒤가 아니라 지금인가?"에 답할 수 있어야 한다.

## **4\. 솔루션 가설**

| 항목 | 내용 |
| ----- | ----- |
| **핵심 가치 제안** | (우리 솔루션이 무엇을 하는가. 1문장) |
| **차별점** | (기존 대안 대비 왜 우리가 나은가) |
| **핵심 가정** | (이것이 맞아야 솔루션이 성립하는 전제) |

**가이드:**

* "핵심 가정"이 가장 중요하다. S2 Pretotype에서 이것을 검증한다  
* 가정은 반증 가능해야 한다. 3개 이내, 가장 위험한 순서로 나열

**예시:**

| 항목 | 내용 |
| ----- | ----- |
| **핵심 가치 제안** | AI가 오답의 근본 원인(수감각)을 분석하여 맞춤 훈련 경로를 제시한다 |
| **차별점** | ① 원인 기반 역동적 난이도 조절 ② 반응시간(유창성)+오답패턴 분석 ③ 게임+공감형 피드백 |
| **핵심 가정** | ① 수감각 훈련이 연산 정확도를 개선한다 (효과) ② 하위 과제 즉각 제시로 아동이 혼자 해결 가능 (자기주도) ③ 공감형 피드백이 학습 지속 시간을 높인다 (리텐션) |

## **5\. 검증 계획**

| 항목 | 내용 |
| ----- | ----- |
| **XYZ 가설** | X% of \[타겟\] will \[행동\] when \[조건\] |
| **핵심 검증 지표** | (어떤 숫자로 판단하는가) |
| **Pretotype 방향** | (S2에서 어떤 실험을 할 것인가) |
| **성공 기준** | (이 숫자 이상이면 Go) |
| **실패 시 대응** | Kill / Pivot 기준 |

**가이드:** "성공 기준"과 "실패 시 대응"을 사전에 정의해야 Gate 판정이 객관적이다. S2 Pretotype Spec에서 상세화된다.

**예시:**

| 항목 | 내용 |
| ----- | ----- |
| **XYZ 가설** | 60% of 수학부진 아동 학부모가 사용자 테스트 후 "도움이 되었다"고 응답한다 |
| **핵심 검증 지표** | 성취도 향상 응답률, 주간 리텐션, NPS |
| **Pretotype 방향** | 웹앱 MVP \+ 타겟 아동·교사 대상 사용자 테스트 |
| **성공 기준** | "도움됨" 60%+ / 주간 리텐션 50%+ / NPS 40%+ |
| **실패 시 대응** | 응답률 30% 미만 → AI 로직 피봇. 리텐션 25% 미만 → UX 전면 재설계 |

## **7\. 일정**

8주(2개월) 총 기간 내에서 각 Stage 시작/종료일을 기입한다. Stage 1에서 러프하게 작성하고, Gate 1 직전에 확정한다.

| \# | Stage | 기간 | 목표일 | Gate | 핵심 의존성 |
| ----- | ----- | ----- | ----- | ----- | ----- |
| **1** | **Brief** | 1W | MM/DD \~ MM/DD | Gate 1: MM/DD |  |
| **2** | **Pretotype** | 2W | MM/DD \~ MM/DD | Gate 2: MM/DD |  |
| **3** | **Spec** | 1W | MM/DD \~ MM/DD | Gate 3: MM/DD |  |
| **4** | **Freeze** | 0.5W | MM/DD \~ MM/DD | Gate 4: MM/DD |  |
| **5** | **MVP Build** | 3.5W | MM/DD \~ MM/DD | Gate 5: MM/DD |  |

**가이드:**

* **핵심 의존성:** Gate 통과에 영향을 주는 외부 요인 (외부 API, 데이터 확보, 인력, 규제 승인 등)  
* S2 Pretotype이 가장 가변적이다. 실험 유형에 따라 1\~3주 변동 가능  
* 전체 8주는 기본값. 프로젝트 특성에 따라 조정하되, Kill Funnel의 속도 원칙을 유지한다

**예시:**

| \# | Stage | 기간 | 목표일 | Gate | 핵심 의존성 |
| ----- | ----- | ----- | ----- | ----- | ----- |
| **1** | **Brief** | 1W | 03/02 \~ 03/06 | Gate 1: 03/06 | \- |
| **2** | **Pretotype** | 2W | 03/09 \~ 03/20 | Gate 2: 03/20 | 타겟 아동 5명 모집 |
| **3** | **Spec** | 1W | 03/23 \~ 03/27 | Gate 3: 03/27 | \- |
| **4** | **Freeze** | 0.5W | 03/30 \~ 04/01 | Gate 4: 04/01 | 디자인 리뷰 |
| **5** | **MVP Build** | 3.5W | 04/02 \~ 04/24 | Gate 5: 04/24 | Web Speech API 검증 |

**Stage 1에서는 여기까지.** §0\~§7만 채우면 나침반으로 기능한다. Gate 1 직전에 §8 보강, 통과 후 §9 병합.

## **AI 인터페이스 참조**

AI가 Brief를 입력으로 다음 산출물을 생성할 때의 매핑. 템플릿 설계 참조용. 실제 Brief 작성 시 삭제.

| 다음 산출물 | Brief에서 읽는 섹션 |
| ----- | ----- |
| **Business Spec** | §2 \+ §3 \+ §4 |
| **Pretotype Spec** | §4 \+ §5 |
| **Product Spec** | §1\~§7 \+ §9 |
| **Design Spec** | §2 \+ §4 \+ §6 |
| **Tech Spec** | §4 \+ §6 \+ Product Spec |

Brief의 빈 칸 \= AI의 환각. 빈 칸이 많을수록 AI 산출물의 정확도가 떨어진다.
