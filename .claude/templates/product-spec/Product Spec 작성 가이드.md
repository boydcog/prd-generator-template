# **Product Spec 작성 가이드 v1.0**

AI가 읽고 코드를 생성할 수 있는 수준의 제품 명세서

* **핵심 질문:** "무엇을 만드는가? 사용자가 핵심 Task를 수행할 수 있는가?"  
* **프로세스 위치:** Stage 3 — Spec (1주)  
* **Gate 연결:** Gate 3 (사용성 \+ 기술검증 \+ AI생성가능성 \+ 프로토타입)  
* **작성:** PM (AI 80% 초안 \+ PM 20% 판단)  
* **데이터 수준:** 검증된 데이터 (Pretotype 실험 결과 \+ 기술 PoC)

## **1\. 기능 개요**

**이 섹션이 답하는 질문:** "이 기능이 왜 존재하는가? 무엇을 하지 않는가?"  
**인풋:** Product Brief §4 (솔루션 가설) \+ Pretotype Spec §5.4 (검증된 핵심 가치)

### **1-1. 기능 목적 & 배경**

| 항목 | 내용 |
| ----- | ----- |
| **기능명** | \[기능/제품명\] |
| **한 줄 목적** | \[이 기능이 해결하는 문제. 1문장\] |
| **배경** | \[왜 이 기능이 필요한가. Pretotype 결과 근거\] |
| **Brief 연결** | \[어떤 Product Brief의 어떤 가설에서 파생되었는가\] |

### **1-2. 기능 우선순위**

| 우선순위 | 기능 | 근거 |
| ----- | ----- | ----- |
| **P0** (MVP 필수) | \[기능\] | \[없으면 핵심 가치 전달 불가\] |
| **P0** | \[기능\] | \[근거\] |
| **P1** (출시 후 빠르게) | \[기능\] | \[근거\] |
| **P2** (이후) | \[기능\] | \[근거\] |

**P0 기준:** Pretotype에서 검증된 핵심 가치를 전달하는 데 반드시 필요한 기능만. P0가 5개를 넘으면 스코프를 재점검한다.

### **1-3. Non-goals (명시적 제외)**

| \# | 하지 않는 것 | 제외 이유 |
| ----- | ----- | ----- |
| 1 | \[기능/범위\] | \[왜 지금은 안 하는가\] |
| 2 | \[기능/범위\] | \[이유\] |

Non-goals는 스코프 관리의 핵심. "안 하는 것"을 명시해야 "하는 것"이 선명해진다.

## **2\. 사용자 스토리**

**이 섹션이 답하는 질문:** "사용자가 무엇을 할 수 있어야 하는가?"  
**형식:** As a \[역할\], I want \[행동\], so that \[가치\]

| ID | 우선순위 | 역할 | 행동 (I want) | 가치 (so that) | 크기 |
| ----- | ----- | ----- | ----- | ----- | ----- |
| US-001 | P0 | \[사용자 역할\] | \[수행할 행동\] | \[얻는 가치/결과\] | S/M/L |
| US-002 | P0 | \[역할\] | \[행동\] | \[가치\] | S/M/L |
| US-003 | P1 | \[역할\] | \[행동\] | \[가치\] | S/M/L |

**좋은 사용자 스토리 체크:**

* 기술 구현이 아닌 **사용자 관점**으로 쓰여 있는가?  
* 하나의 스토리가 **하나의 가치**를 전달하는가?  
* **테스트 가능**한가? (→ §4 완료 기준과 1:1 매핑)  
* 크기(S/M/L)는 **1주 내 구현 가능**한 수준인가? L이면 분할을 검토한다.

## **3\. 사용자 흐름**

**이 섹션이 답하는 질문:** "사용자가 어떤 경로로 움직이는가?"  
**인풋:** §2 사용자 스토리의 핵심 시나리오

### **3-1. Happy Path (핵심 성공 경로)**

P0 사용자 스토리의 이상적 수행 경로. 가장 중요한 1\~2개 흐름.

| \[시작점\] → \[행동 1\] → \[행동 2\] → \[행동 3\] → \[성공 상태\] |
| :---- |

| Step | 화면/상태 | 사용자 행동 | 시스템 반응 |
| ----- | ----- | ----- | ----- |
| 1 | \[화면명\] | \[사용자가 하는 것\] | \[시스템이 보여주는 것\] |
| 2 | \[화면명\] | \[행동\] | \[반응\] |
| 3 | \[화면명\] | \[행동\] | \[반응\] |

### **3-2. Alternative Path (대안 경로)**

성공하지만 다른 경로를 거치는 시나리오.

| 시나리오 | 분기점 | 대안 경로 | 합류점 |
| ----- | ----- | ----- | ----- |
| \[시나리오명\] | \[어디서 갈라지는가\] | \[다른 경로\] | \[어디서 합류하는가\] |

### **3-3. Error Flow (에러 처리 경로)**

실패 시 사용자에게 어떤 경험을 제공하는가.

| 에러 상황 | 발생 지점 | 사용자에게 보이는 것 | 복구 경로 |
| ----- | ----- | ----- | ----- |
| \[에러명\] | \[어느 Step에서\] | \[에러 메시지/UI\] | \[어떻게 돌아오는가\] |

## **4\. 완료 기준 (Acceptance Criteria)**

**이 섹션이 답하는 질문:** "언제 '완료'인가?"  
**형식:** Given/When/Then (BDD) 또는 체크리스트. 스토리별 1:1 매핑.

### **Given/When/Then 형식**

| 스토리 ID | Given (전제) | When (행동) | Then (결과) |
| ----- | ----- | ----- | ----- |
| US-001 | \[사전 조건\] | \[사용자 행동\] | \[예상 결과\] |
| US-001 | \[다른 전제\] | \[행동\] | \[결과\] |
| US-002 | \[전제\] | \[행동\] | \[결과\] |

### **또는 체크리스트 형식**

**US-001: \[스토리 제목\]**

- [ ] \[검증 가능한 조건 1\]  
- [ ] \[검증 가능한 조건 2\]  
- [ ] \[검증 가능한 조건 3\]

**AC 품질 기준:**

* 모든 P0 스토리에 AC가 있는가?  
* AC가 **관찰/측정 가능**한가? ("잘 작동한다" ✗ → "3초 내 결과 표시" ✓)  
* 개발자가 AC만 읽고 **테스트 코드를 작성**할 수 있는가?

## **5\. 엣지 케이스 & 제약**

**이 섹션이 답하는 질문:** "경계 조건에서 무슨 일이 벌어지는가?"

| 유형 | 시나리오 | 예상 동작 | 우선순위 |
| ----- | ----- | ----- | ----- |
| **입력 제한** | \[최소/최대값, 빈 입력, 특수문자 등\] | \[시스템 반응\] | P0/P1 |
| **동시성** | \[다중 사용자 동시 접근/수정\] | \[시스템 반응\] | P0/P1 |
| **권한** | \[비인가 접근, 역할 초과 행동\] | \[시스템 반응\] | P0/P1 |
| **네트워크** | \[오프라인, 타임아웃, 느린 연결\] | \[시스템 반응\] | P1/P2 |
| **데이터** | \[빈 데이터, 대량 데이터, 잘못된 형식\] | \[시스템 반응\] | P0/P1 |

모든 엣지 케이스를 나열할 필요 없다. **P0 기능의 핵심 경계 조건**만 다룬다. 나머지는 개발 시 Tech Spec에서 확장.

## **6\. 데이터 요구사항**

**이 섹션이 답하는 질문:** "어떤 데이터가 필요한가?"  
**이 섹션이 Tech Spec의 시작점이 된다.**

| 데이터 항목 | 소스 | 포맷 | 갱신 주기 | 의존성 |
| ----- | ----- | ----- | ----- | ----- |
| \[데이터명\] | \[어디서 오는가\] | \[JSON/CSV/DB 등\] | \[실시간/일배치/수동\] | \[다른 시스템/API\] |
| \[데이터명\] | \[소스\] | \[포맷\] | \[주기\] | \[의존성\] |

### **Input/Output 명세**

| 기능 | Input | Output | 비고 |
| ----- | ----- | ----- | ----- |
| \[기능명\] | \[사용자가 입력하는 것\] | \[시스템이 출력하는 것\] | \[제약/조건\] |

## **7\. 디자인 프롬프트**

**이 섹션이 답하는 질문:** "UI는 어떤 규칙을 따르는가?"  
**Stage 3에서 작성, Stage 4 Freeze 시 Design Spec으로 분리·확장된다.** AI가 이 규칙을 읽고 UI를 생성한다. 시안이 아닌 **규칙**을 기술한다.

### **7-1. 정보 구조 (IA)**

| 화면 | 상위 | 접근 경로 | 핵심 콘텐츠 |
| ----- | ----- | ----- | ----- |
| \[화면명\] | \[상위 화면\] | \[네비게이션 경로\] | \[화면의 핵심 요소\] |

### **7-2. 컴포넌트 매핑**

| UI 요소 | 표준 컴포넌트 | 커스텀 여부 | 비고 |
| ----- | ----- | ----- | ----- |
| \[요소명\] | \[shadcn/Button 등\] | 표준/커스텀 | \[특이사항\] |

**원칙:** 표준 라이브러리(shadcn 등) 컴포넌트를 최대한 활용. 커스텀은 최소화.

### **7-3. 레이아웃 규칙**

| 항목 | 규칙 |
| ----- | ----- |
| **Grid** | \[컬럼 수, 거터\] |
| **Spacing** | \[기본 단위, 여백 규칙\] |
| **Responsive** | \[브레이크포인트, 모바일 대응\] |

### **7-4. 디자인 토큰**

| 토큰 | 값 | 용도 |
| ----- | ----- | ----- |
| **Primary Color** | \[\#hex\] | \[주요 CTA, 강조\] |
| **Background** | \[\#hex\] | \[배경\] |
| **Typography** | \[폰트/사이즈\] | \[본문/제목\] |
| **Spacing Unit** | \[px/rem\] | \[기본 간격\] |

**자가 점검:** AI(또는 디자이너)가 이 섹션만 읽고 UI를 생성할 수 있는가? 가능하면 이 섹션의 역할을 다한 것이다. 불가능하면 규칙이 부족하다.

## **Gate 3 자가 점검**

Product Spec 완성 후, Gate 3 통과 전에 자가 점검한다.

| 기준 | 질문 | 상태 |
| ----- | ----- | ----- |
| **사용성** | 사용자가 도움 없이 핵심 Task를 수행할 수 있는가? (프로토타입 테스트) | \[Pass/Fail\] |
| **기술 검증** | 핵심 기술 PoC가 통과했는가? 4주 내 구현 가능한가? | \[Pass/Fail\] |
| **AI 생성 가능성** | AI가 이 Spec만 읽고 UI/코드를 생성할 수 있는가? | \[Pass/Fail\] |
| **프로토타입** | 핵심 흐름이 작동하는 프로토타입이 존재하는가? | \[Pass/Fail\] |

4개 모두 Pass → Gate 3 Go. 1개라도 Fail → 해당 항목 보완 후 재점검.

## **작성 가이드**

### **PM → AI 워크플로우**

| Step | 누가 | 시간 | 행동 |
| ----- | ----- | ----- | ----- |
| 1 | PM | 30분 | Brief §4\~§6 \+ Pretotype §5.4에서 검증된 기능·가치·스코프 추출. 경로 선택 (A/B). |
| 2 | AI (Planner) | 30분 | Product Spec 초안 (§1\~§6). Brief→Spec 변환. |
| 3 | AI (Critic) | 20분 | 사용성 리스크 \+ 기술 리스크 5가지 공격. |
| 4 | PM | 30분 | Critic 피드백 반영. 엔지니어와 Technical Feasibility 확인. |
| 5 | AI (Editor) | 15분 | 최종 수정 \+ 품질 점수 (8점 이상 통과, 미만 시 Step 2 회귀). |
| 6 | PM \+ 디자이너 | 30분 | §7 디자인 프롬프트 공동 작성. |
| 7 | AI 코딩 도구 | 1\~2일 | §1\~§7 기반 프로토타입 생성 (Lovable/v0/Cursor). |
| 8 | PM | 반나절 | 프로토타입 검증 \+ Gate 3 자가 점검. |

**루프:** Step 2→3→5를 최대 3회 반복. 8점 이상이면 통과.

### **인풋 체크리스트**

- [ ] Product Brief (Gate 1 통과본) \- §4 솔루션 가설, §5 검증 계획, §6 스코프  
- [ ] Pretotype Spec (Gate 2 통과본) \- §5.4 Go 시 인풋 요약  
- [ ] Business Spec \- §4 수익 구조 (과금이 기능에 영향을 줄 경우)  
- [ ] 엔지니어 Technical Feasibility 확인 (Stage 3 착수 전 또는 중)  
- [ ] 디자인 컴포넌트 라이브러리 확인 (shadcn 등 어떤 표준을 쓰는가)

### **경로 선택 \- A: PM 바이브코딩 vs B: Spec→Tech 직행**

| 항목 | 경로 A: PM 바이브코딩 | 경로 B: Spec→Tech 직행 |
| ----- | ----- | ----- |
| **적합** | 단순 기능, UI 중심, 빠른 가설 검증 | 복잡 로직, 기존 시스템 연동, 높은 기술 난이도 |
| **Stage 3 실행** | PM이 Lovable/v0로 프로토타입 직접 제작 | Tech팀이 Cursor 등으로 프로토타입 구현 |
| **Stage 3 문서** | Brief \+ Product Spec (2종) | Brief \+ Product Spec \+ Tech Spec 경량 (3종) |
| **Stage 4** | 프로토타입 \+ Frozen Spec → Tech팀 인수 | Frozen Spec 기반 → Tech팀 빌드 |

**판단 기준:** "PM이 AI 코딩 도구만으로 핵심 흐름을 시연할 수 있는가?" Yes → 경로 A. No → 경로 B.

### **Stage별 작성 순서**

**Stage 3 시작 시 (경량 \- 반나절):**

1. §1 기능 개요 (P0 기능 목록 \+ Non-goals)  
2. §2 사용자 스토리 (P0 스토리만)  
3. §4 완료 기준 (P0 AC만)

**Stage 3 진행하면서 (완성 \- 2\~3일):** 4\. §3 사용자 흐름 (Happy Path 필수, 나머지 주요 경로) 5\. §5 엣지 케이스 (P0 기능의 핵심 경계만) 6\. §6 데이터 요구사항 7\. §7 디자인 프롬프트 (디자이너 협업) 8\. 프로토타입 생성 \+ 검증

**Stage 4 (확정 \- Gate 3 통과 후):** 9\. §7 → Design Spec으로 분리 (디자이너) 10\. §5, §6 → Tech Spec 초안 확장 (엔지니어) 11\. 전 섹션 Freeze

### **복잡 프로덕트 확장**

기능 영역(Epic)이 3개 이상이면 Product Spec을 분리한다.

| 프로덕트 (Brief 1개) ├── 기능 영역 A → Product Spec A ├── 기능 영역 B → Product Spec B └── 기능 영역 C → Product Spec C    └── Design Spec 1개 (프로덕트 공유)    └── Tech Spec A, B, C (기능 영역별) |
| :---- |

Design Spec은 **규칙**이므로 프로덕트 전체에 1개면 충분하다.

### **자가진단**

* **사람 테스트:** 처음 참여하는 개발자가 이 Spec만 읽고 2분 내에 "이 기능이 뭘 하는지" 설명할 수 있는가?  
* **AI 테스트:** 이 Spec을 AI 코딩 도구에 넣으면 핵심 흐름이 작동하는 프로토타입을 생성하는가?  
* **Freeze 테스트:** 엔지니어가 추가 질문 없이 개발을 시작할 수 있는가?
