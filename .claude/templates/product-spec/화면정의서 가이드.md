# 화면정의서 가이드 v1.0

AI가 화면별 콘텐츠 명세를 생성할 때 참조하는 기준서.
Product Spec §3-2 화면 목록의 각 화면(SCR-N)을 이 10개 항목으로 상세화한다.

---

## Part 1: 프레임워크 개요

### 화면 정의 10개 항목

| # | 항목 | 한 줄 설명 |
|---|------|-----------|
| 1 | **User Goal** | 이 화면에서 사용자가 달성하려는 목표 |
| 2 | **System Goal** | 시스템이 이 화면을 통해 달성하려는 목표 |
| 3 | **Data Model** | 화면에 표시되거나 입력받는 데이터 구조 |
| 4 | **Information Hierarchy** | 화면 내 정보의 우선순위와 배치 순서 |
| 5 | **UI Components** | 화면을 구성하는 컴포넌트 목록 |
| 6 | **Interactions** | 사용자 액션과 시스템 반응 (탭, 스와이프, 입력 등) |
| 7 | **Error States** | 에러 발생 시 보여줄 상태와 메시지 |
| 8 | **Empty States** | 데이터가 없을 때 보여줄 상태 |
| 9 | **Accessibility** | 접근성 요구사항 (스크린리더, 대비율, 터치 영역 등) |
| 10 | **Constraints** | 기술적·비즈니스적 제약 (성능, 권한, 플랫폼 등) |

**원칙:**
- 모든 P0 화면은 10개 항목을 모두 채운다.
- P1/P2 화면은 최소 1~5번 항목 필수, 6~10번은 선택.
- AI가 이 명세만 읽고 컴포넌트 코드를 생성할 수 있는 수준으로 작성한다.

## Part 2: 항목별 작성 가이드

### 2.1 User Goal

**무엇을:** 사용자가 이 화면에 진입한 이유. 단 하나의 핵심 목표를 동사형으로 서술한다.

**어떻게:** `~한다` 형태. 사용자 스토리(US-N)와 연결한다.

**분량:** 1~2문장.

**작성 팁:**
- "정보를 본다"보다 구체적으로 → "오늘의 운동 기록을 확인하고 다음 세션을 예약한다"
- 하위 목표가 있으면 Primary Goal / Secondary Goal로 구분한다

**예시:**
```
Primary: 오늘의 수학 문제를 풀고 오답을 즉시 확인한다 (US-001)
Secondary: 이전 오답 패턴을 참고해 풀이 전략을 선택한다
```

---

### 2.2 System Goal

**무엇을:** 시스템(서비스) 관점에서 이 화면이 달성해야 하는 목표. 비즈니스/학습/참여 지표로 표현한다.

**어떻게:** 측정 가능한 지표와 연결한다.

**분량:** 1~3항목.

**작성 팁:**
- 단순 "화면 표시"가 아닌 서비스 목표 연결 → "리텐션을 위해 오늘의 진도 완료 경험 제공"
- KPI가 있으면 명시한다 (예: 세션 완료율 80% 이상)

**예시:**
```
- 문제 풀이 완료율 80% 이상 유도 (리텐션 지표)
- 오답 발생 시 즉시 피드백으로 이탈 방지
- 다음 세션 예약 전환율 30% 이상
```

---

### 2.3 Data Model

**무엇을:** 화면에서 읽거나 쓰는 데이터 구조. 필드명, 타입, 출처, 필수 여부를 정의한다.

**어떻게:** 표 형식 또는 JSON 스키마 스케치.

**분량:** 핵심 필드만. Tech Spec §4에서 상세화한다.

**작성 팁:**
- 표시용(read)과 입력용(write)을 구분한다
- 외부 API에서 오는 데이터는 출처를 명시한다
- 민감 데이터(개인정보 등)는 마스킹 여부를 표기한다

**예시:**
```
| 필드 | 타입 | 출처 | 필수 | 비고 |
|------|------|------|------|------|
| problem_id | string | DB | ✓ | 오늘의 문제 |
| problem_text | string | DB | ✓ | LaTeX 렌더링 |
| user_answer | string | 입력 | ✓ | 최대 500자 |
| is_correct | boolean | 서버 판정 | — | 제출 후 반환 |
| hint_count | int | DB | — | 기본값 0 |
```

---

### 2.4 Information Hierarchy

**무엇을:** 화면 내 정보 요소의 시각적 우선순위와 배치 논리.

**어떻게:** 1순위부터 나열하거나, 영역(Zone)으로 구분한다.

**분량:** 3~7개 요소.

**작성 팁:**
- F-패턴(웹) 또는 상단 집중(모바일)을 고려한다
- 사용자 목표 달성에 가장 필요한 정보가 1순위
- 보조 정보(메타데이터, 날짜, 카운터 등)는 후순위

**예시:**
```
1순위: 문제 텍스트 (화면 중앙 상단, 가장 큰 글씨)
2순위: 입력 영역 (문제 바로 아래)
3순위: 힌트 버튼 (입력 영역 옆, 보조)
4순위: 문제 번호 / 진도 표시 (상단 바)
5순위: 이전 오답 참고 링크 (하단, 선택적 노출)
```

---

### 2.5 UI Components

**무엇을:** 화면을 구성하는 UI 컴포넌트 목록. 컴포넌트명, 상태, 연결 액션을 정의한다.

**어떻게:** 표 형식.

**분량:** 화면의 모든 상호작용 요소 포함.

**작성 팁:**
- 디자인 시스템 컴포넌트명을 사용한다 (예: Button.Primary, Input.Text)
- 컴포넌트별 상태(default/disabled/loading/error)를 명시한다
- 커스텀 컴포넌트는 역할을 설명한다

**예시:**
```
| 컴포넌트 | 역할 | 상태 | 트리거 액션 |
|---------|------|------|------------|
| ProblemCard | 문제 표시 | — | — |
| Input.Text | 답안 입력 | default / error | onChange → validate |
| Button.Primary | 제출 | default / loading / disabled | onSubmit |
| HintButton | 힌트 요청 | default / used | onClick → hint modal |
| ProgressBar | 진도 표시 | — | — |
```

---

### 2.6 Interactions

**무엇을:** 사용자 액션 → 시스템 반응의 쌍. 화면 전환, 상태 변화, 애니메이션을 포함한다.

**어떻게:** 트리거 → 반응 형식.

**분량:** 주요 인터랙션만. 엣지 케이스는 §7(에러 상태)에서 다룬다.

**작성 팁:**
- 응답 시간을 명시한다 (즉각 / 300ms / 2초 이내)
- 로딩 상태가 필요한 액션은 별도 표기한다
- 화면 전환 시 이동하는 화면 ID(SCR-N)를 명시한다

**예시:**
```
| 트리거 | 조건 | 반응 | 전환 화면 |
|--------|------|------|----------|
| 제출 버튼 탭 | 답안 입력됨 | 로딩(300ms) → 정답/오답 피드백 표시 | — (인라인) |
| 제출 버튼 탭 | 답안 미입력 | 입력 필드 강조 + "답을 입력해주세요" | — |
| 정답 확인 후 다음 버튼 탭 | — | 다음 문제로 전환 (슬라이드 애니메이션) | SCR-003 |
| 힌트 버튼 탭 | hint_count < 3 | 힌트 모달 오픈 | — (모달) |
| 뒤로가기 | — | 이탈 확인 다이얼로그 | SCR-001 (확인 시) |
```

---

### 2.7 Error States

**무엇을:** 에러 발생 시 사용자에게 보여주는 상태와 복구 경로.

**어떻게:** 에러 유형별로 메시지, UI 변화, 복구 방법을 정의한다.

**분량:** P0 기능의 핵심 에러만. 상세 에러는 Tech Spec §6으로 이관.

**작성 팁:**
- 사용자 언어로 작성한다 (기술 오류 코드 금지)
- 복구 액션을 항상 제공한다 (재시도, 돌아가기 등)
- 에러 메시지는 원인 + 해결 방법 포함

**예시:**
```
| 에러 유형 | 발생 조건 | 메시지 | UI 변화 | 복구 액션 |
|----------|----------|--------|---------|----------|
| 입력 미완성 | 빈 답안 제출 | "답을 입력해주세요" | 입력 필드 빨간 테두리 | 입력 후 재제출 |
| 네트워크 오류 | 제출 중 연결 끊김 | "연결이 끊겼습니다. 다시 시도해주세요" | 재시도 버튼 표시 | 재시도 버튼 탭 |
| 세션 만료 | 토큰 만료 | "세션이 만료되었습니다" | 로그인 화면 이동 버튼 | SCR-LOGIN으로 이동 |
```

---

### 2.8 Empty States

**무엇을:** 표시할 데이터가 없을 때 보여주는 화면 상태.

**어떻게:** 빈 상태 유형별로 메시지와 CTA를 정의한다.

**분량:** 해당 화면에서 발생 가능한 빈 상태만.

**작성 팁:**
- "데이터 없음" 대신 맥락에 맞는 메시지 (첫 방문 vs. 필터 결과 없음)
- 항상 다음 행동(CTA)을 제공한다
- 일러스트/아이콘이 있으면 설명한다

**예시:**
```
| 빈 상태 유형 | 발생 조건 | 메시지 | CTA |
|------------|----------|--------|-----|
| 첫 방문 | 학습 기록 없음 | "아직 푼 문제가 없어요. 첫 문제를 시작해볼까요?" | "문제 시작하기" 버튼 |
| 오늘 완료 | 오늘치 문제 모두 완료 | "오늘 학습 완료! 내일 또 만나요 🎉" | "오답 노트 보기" 버튼 |
| 필터 결과 없음 | 검색/필터 적용 후 | "해당하는 문제가 없어요. 필터를 바꿔보세요." | "필터 초기화" 버튼 |
```

---

### 2.9 Accessibility

**무엇을:** 이 화면의 접근성 요구사항.

**어떻게:** WCAG 기준 + 플랫폼 가이드라인을 체크리스트로 정리.

**분량:** 화면에 적용되는 항목만.

**작성 팁:**
- 스크린리더 레이블은 기능적으로 설명 (아이콘만 있는 버튼에 필수)
- 색상만으로 의미를 전달하지 않는다 (정답/오답은 색 + 아이콘 + 텍스트)
- 타겟 사용자가 고령자/장애인이면 기준을 높인다 (WCAG AAA)

**예시:**
```
| 항목 | 기준 | 적용 내용 |
|------|------|---------|
| 색상 대비 | WCAG AA (4.5:1) | 텍스트 #1A1A1A / 배경 #FFFFFF |
| 터치 영역 | 최소 44×44px | 힌트 버튼, 제출 버튼 |
| 스크린리더 | aria-label 필수 | 힌트 버튼: "힌트 보기 (남은 횟수: 2회)" |
| 포커스 순서 | 논리적 순서 | 문제 → 입력 → 힌트 → 제출 |
| 오류 알림 | 에러 시 focus 이동 | 에러 발생 시 해당 필드로 포커스 이동 |
```

---

### 2.10 Constraints

**무엇을:** 이 화면의 기술적·비즈니스적 제약.

**어떻게:** 제약 유형별로 조건과 영향을 명시한다.

**분량:** 실제로 구현에 영향을 주는 제약만.

**작성 팁:**
- "빠르게" 같은 모호한 표현 대신 수치로 → "3초 이내 로딩"
- 플랫폼 제약(iOS/Android 차이)이 있으면 명시
- 권한 제약(로그인 필수, 구독 필요 등)을 포함

**예시:**
```
| 제약 유형 | 조건 | 영향 |
|---------|------|------|
| 성능 | 문제 로딩 3초 이내 | 스켈레톤 로더 필수 |
| 오프라인 | 인터넷 없으면 접근 불가 | 오프라인 안내 화면 필요 |
| 권한 | 로그인 필수 | 미로그인 시 SCR-LOGIN 리다이렉트 |
| 구독 | P1 기능은 유료 구독자만 | 무료 사용자에게 업그레이드 CTA 표시 |
| LaTeX | 수식 렌더링 라이브러리 필요 | KaTeX 또는 MathJax 의존성 |
```
