# **Business Spec 작성 가이드 v1.0**

제품 아이디어의 사업 타당성을 가설 수준에서 구조적으로 평가하는 문서.

* **핵심 질문:** "이 아이디어에 2주(Pretotype)를 투자할 가치가 있는가?"  
* **기반:** Lean Canvas \+ bREAD Market Opportunity/Business Viability 레이어.  
* **위치:** Stage 1 Brief와 병행 작성. Brief §1(시장&사업성)의 근거 문서.  
* **Gate:** Gate 1(시장+사업성) Go/No-Go 판정의 핵심 입력물.  
* **작성자:** PM (AI 80% 초안 \+ PM 20% 판단)  
* **데이터 수준:** 2차 데이터(보고서, 통계, 커뮤니티) \+ 벤치마크 추정. 1차 데이터(실험 결과)는 Stage 2 Pretotype에서 수집.

## **문서 관계**

| Business Spec ──근거──→ Brief §1(시장&사업성)      │                         │      └───── 합산 ─────→ Gate 1 Go/No-Go 판정                                 │                          Stage 2 Pretotype ← §6 XYZ 가설 전달 |
| :---- |

* **Brief**: 왜 만드는가 \+ 누구를 위해 \+ 무엇을 (제품 관점)  
* **Business Spec**: 돈을 벌 수 있는 구조인가 (사업 관점)  
* Brief 없이 Business Spec만으로 Gate 1을 통과할 수 없고, 그 반대도 마찬가지다.  
* §6의 XYZ 가설과 Pretotype KPI는 Stage 2 Pretotype Report의 입력물이 된다.

## **템플릿 구조 \- 8섹션**

### **1\. 사업 가설 & 가치 제안**

#### **사업 가설 (1문장)**

**형식:** "\[타겟\]의 \[문제\]를 \[솔루션\]으로 해결하여 \[수익모델\]로 \[목표시장\]에서 수익을 만든다."

#### **UVP \- 고유 가치 제안 (1문장)**

**형식:** 고객이 수많은 대안 중 왜 반드시 우리를 써야 하는지. 기능이 아닌 고객이 얻는 결과와 정서적 이득.

#### **P0 솔루션 (3\~5개)**

이 가치 제안을 실현하는 핵심 기능. Must-Have만. Nice-to-Have는 적지 않는다.

| \# | P0 기능 | 해결하는 문제 |
| ----- | ----- | ----- |
| 1 |  |  |
| 2 |  |  |
| 3 |  |  |

**작성 기준:**

* 사업 가설은 Business Spec 전체의 요약. §2\~§7은 이 가설의 근거.  
* UVP는 "우리가 잘하는 것"이 아니라 "고객이 얻는 것"으로 쓴다.  
* P0은 5개 초과 금지. 초과하면 핵심이 없다는 뜻이다.

### **2\. 시장 기회**

| 항목 | 내용 |
| ----- | ----- |
| **TAM** | 산업 전체 규모 (금액 \+ 출처 \+ 연도) |
| **SAM** | 비즈니스 모델로 5년 내 도달 가능 범위 (지역/세그먼트 한정) |
| **SOM** | MVP 출시 후 2년 내 점유 가능한 구체적 목표 (고객 수 × ARPU) |
| **성장 동인** | 시장이 커지는 이유 3가지 |
| **타이밍** | 왜 지금인가? (규제/기술/행동 변화) |

**작성 기준:**

* 숫자 없는 시장 분석은 분석이 아니다. 모든 항목에 금액/비율/출처 필수.  
* TAM은 희망이 아니라 팩트. Top-down \+ Bottom-up 교차 검증 권장.  
* SOM이 가장 중요. "2년 내 우리가 실제로 먹을 수 있는 크기"를 보수적으로.

**AI 생성 가이드:**

* 인풋: 산업 보고서, 통계청 데이터, 경쟁사 매출, 타겟 인구  
* AI에게 "TAM/SAM/SOM을 bottom-up으로 산출하고 출처를 명시하라" 지시

### **3\. 고객 & 고통**

#### **3-1. 핵심 고객 세그먼트**

| 세그먼트 | 규모 | 현재 지출 | 고통 강도 |
| ----- | ----- | ----- | ----- |
| 1차 타겟 | 명/사 | 연 ₩ | Painkiller / Vitamin |
| 2차 타겟 | 명/사 | 연 ₩ | Painkiller / Vitamin |

**얼리어답터:** 핵심 가치를 가장 절실히 원하는 구체적 페르소나 1명을 정의한다. "누구인가"가 아니라 "어떤 문제를 겪고 있는 누구인가"에 집중.

#### **3-2. Painkiller 검증 (NPQ 테스트)**

| 질문 | 답변 | 점수 |
| ----- | ----- | ----- |
| **N**eed: 지금 이 문제를 해결하려고 능동적으로 행동하는가? |  | /3 |
| **P**ay: 현재 유사 솔루션에 돈을 쓰고 있는가? (금액?) |  | /3 |
| **Q**uantify: 미해결 시 손실을 숫자로 말할 수 있는가? |  | /3 |

NPQ 합계 7+ \= Painkiller 확정. 5\~6 \= 조건부. 4- \= Vitamin.

#### **3-3. Skin in the Game \- 시장의 기존 행동 증거**

고객의 말이 아닌 행동으로 고통 강도를 측정. **Stage 1**에서는, 이 시장 고객이 이미 하고 있는 행동 (2차 데이터: 커뮤니티, 보고서, 경쟁사 이용 데이터) **Stage 2** 업데이트 시, 우리 프리토타입에 대한 실제 반응 (1차 데이터: 실험 결과)

| 점수 | 행동 수준 | 시장 기존 행동 증거  (Stage 1\) | 우리 실험 결과  (Stage 2 업데이트) |
| ----- | ----- | ----- | :---: |
| 0 | 의견 |  | \- |
| 1 | 이메일/알림 신청 |  | \- |
| 10 | 개인정보 제공 |  | \- |
| 30 | 시간/노동 투자 |  | \- |
| 100 | 결정적 증거 (결제, 직접 방문) |  | \- |

**Stage 1 최고 점수: \_\_\_점** (시장 기존 행동) 30점 이상이면 시장 수요 강력. 100점이면 확실.

#### **3-4. 지불의사 근거**

* 현재 대안에 쓰는 비용: ₩\_\_\_/월  
* 우리 솔루션의 기대 가격: ₩\_\_\_/월  
* 가격 대비 가치 논리: (비용 절감? 시간 절감? 리스크 제거?)

**작성 기준:**

* "고객이 아프다"는 주장만으로 부족. 고통을 숫자로 환산 (시간, 비용, 기회비용).  
* Painkiller/Vitamin 분류는 감이 아니라 NPQ 점수 \+ Skin in the Game 행동 증거.  
* 사용자의 "말"이 아닌 "행동"에 집착.

### **4\. 수익 구조**

#### **4-1. Revenue Model**

| 구분 | 내용 |
| ----- | ----- |
| **과금 방식** | 구독 / 건당 결제 / 프리미엄 / 라이선스 / 광고 / 하이브리드 |
| **가격** | ₩\_\_\_/월 (근거: 경쟁사 가격 ± 차별화 프리미엄) |
| **과금 단위** | 사용자당 / 팀당 / 사용량 기반 |
| **Upsell 경로** | 무료→유료 / 기본→프리미엄 / B2C→B2B |

#### **4-2. Unit Economics (추정)**

| 지표 | 목표 | 산출 근거 |
| ----- | ----- | ----- |
| **ARPU** | ₩\_\_\_/월 | 가격 × 예상 과금율 |
| **CAC** | ₩\_\_\_ | 채널별 획득 비용 추정 |
| **LTV** | ₩\_\_\_ | ARPU × 평균 이용 기간(월) |
| **LTV/CAC** | \_\_\_x | **3x 이상 필수** (황금비율) |
| **Payback** | \_\_\_개월 | CAC ÷ 월 마진 |
| **Gross Margin** | \_\_\_% | (매출 \- 변동비) ÷ 매출 |

**작성 기준:**

* MVP 단계에서 정밀한 Unit Economics는 불가능. 가정을 명시하고, 검증 방법을 함께 기재.  
* LTV/CAC 3x 미만이면 사업모델을 재검토.  
* "추후 확인" 금지. 가정이라도 숫자 기재.

### **8\. Go/No-Go 판정**

#### **스코어카드**

| \# | 평가 항목 | 질문 | 점수 (1-5) | 기준 |
| ----- | ----- | ----- | ----- | ----- |
| 1 | **시장 규모** | SAM이 충분한가? | /5 | 1=\<10억 / 3=50\~200억 / 5=\>500억 |
| 2 | **고통 강도** | Painkiller인가? | /5 | 1=Vitamin / 3=강한Vitamin / 5=NPQ 7+ Painkiller |
| 3 | **수익성** | LTV/CAC 3x 달성 가능한가? | /5 | 1=\<1x / 3=2\~3x / 5=\>5x |
| 4 | **경쟁 우위** | Unfair Advantage가 있는가? | /5 | 1=없음 / 3=시간우위 / 5=구조적 Moat |
| 5 | **실행 적합성** | 우리 팀이 만들 수 있는가? | /5 | 1=역량 부족 / 3=학습 필요 / 5=핵심 역량 |

#### **판정 기준**

이 판정은 "사업 확신"이 아니라 "Pretotype 투자 가치"를 판단한다. 모든 점수는 2차 데이터와 가설 기반. Stage 2 Pretotype에서 1차 데이터로 재검증한다.

| 총점 | 판정 | 의미 |
| ----- | ----- | ----- |
| **20\~25** | **Go** | 가설이 충분히 타당. 2주 Pretotype 투자 가치 있음. Stage 2 진입. |
| **15\~19** | **Conditional Go** | 가설에 약한 고리 있음. 특정 항목 보완 후 진입. 조건 명시 필수. |
| **\~14** | **Kill** | 가설 수준에서도 사업성 불분명. 재설계 또는 폐기. |

### **자가진단**

* 사람 테스트: 이 문서만으로 2분 내에 "이 제품의 사업 타당성"을 설명할 수 있는가?  
* AI 테스트: AI가 이 문서를 읽고 Pretotype Spec의 검증 설계를 올바르게 생성하는가?

### **업데이트 주기**

| 시점 | 행동 |
| ----- | ----- |
| Stage 1 (Brief) | 초판 작성 \- 가설 기반 |
| Stage 2 (Pretotype) | 검증 반영 \- 실험 데이터로 §3 Skin in the Game, §4, §6 XYZ 결과 업데이트 |
| Gate 2 (Kill Gate) | Go/No-Go 재판정 \- 실제 데이터 기반 최종 판단 |
| Stage 3+ | 참조만. 추가 업데이트 없음. |

### **AI 인터페이스 참조**

AI가 Business Spec을 입력으로 다음 산출물을 생성할 때의 매핑. 템플릿 설계 참조용. 실제 작성 시 삭제.

| 다음 산출물 | Business Spec에서 읽는 섹션 |
| ----- | ----- |
| Pretotype Spec | §1 \+ §3 \+ §6 (가설, 고객 고통, 검증 설계) |
| Product Spec | §1 \+ §2 \+ §4 \+ §5 (가치제안, 시장, 수익, 경쟁우위) |
| Brief §8 병합 | §2 \+ §4 (시장 규모, 수익 구조) |

Business Spec의 빈 칸 \= AI의 환각. 빈 칸이 많을수록 AI 산출물의 정확도가 떨어진다.
